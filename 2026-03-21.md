# GPU 加速方案讨论记录 — 2026-03-21

> Phase 2e: 将 ABA / FK / Contact / Integration 移植到 GPU，目标 N=1000+ 并行环境。

---

## 1. 当前 CPU 算法结构

一步 `Simulator.step()` 的计算流：

```
passive_torques(q, qdot)          # 遍历关节，逐个算限位+阻尼
    |
FK: forward_kinematics(q)        # root->leaf 前向递推，逐 body 矩阵乘
    |
body_velocities(q, qdot)         # root->leaf 前向递推（与 FK 冗余一半）
    |
contact.compute_forces(X, v)     # 遍历 contact points，独立计算
    |
collision.compute_forces(X, v)   # 遍历 AABB pairs，独立计算
    |
ABA(q, qdot, tau+tau_passive, ext_forces)  # 3-pass Featherstone
    |
integrator.step -> qdot_new, q_new         # 向量加法 + 四元数积分
```

### 各模块并行性

| 模块 | 单 env 内并行性 | 跨 env 并行性 | 计算量占比 |
|------|----------------|---------------|-----------|
| ABA (3-pass) | 链式依赖（parent<->child） | 完全独立 | ~70% |
| FK | 同上 | 完全独立 | 中 |
| body_velocities | 同上 | 完全独立 | 中 |
| passive_torques | 逐关节独立 | 完全独立 | 低 |
| contact forces | 逐点独立 | 完全独立 | 低-中 |
| self-collision | 逐 pair 独立 | 完全独立 | 低 |
| integrator | 逐 DOF 独立 | 完全独立 | 低 |

### ABA 的链式依赖

ABA 是 Featherstone 的 O(n) 算法，3 个 pass：

- **Pass 1 (forward, root->leaf)**：计算每个 body 的速度 v[i]、bias c[i]、初始 IA[i]、pA[i]。每个 body 依赖 parent。
- **Pass 2 (backward, leaf->root)**：累积关节空间惯量 IA[parent] += X^T @ IA_A @ X。每个 body 必须等所有 children 先完成。
- **Pass 3 (forward, root->leaf)**：计算加速度 a[i] 和 qddot[i]。依赖 a[parent]。

四足机器人树深度约 L=5，每层 body 数为 [1, 4, 4, 4, 4]，总 n=17 个 body。

## 2. 数据表示

### Build 时固定（所有 env 共享，只读）

```
parent[i]     : (n,)       int     拓扑
joint_type[i] : (n,)       int     {0=rev, 1=pris, 2=fixed, 3=free}
axis[i]       : (n, 3)     float   关节轴
X_tree_R[i]   : (n, 3, 3)  float   树变换旋转
X_tree_r[i]   : (n, 3)     float   树变换平移
I_mat[i]      : (n, 6, 6)  float   空间惯量（预算好，含 CoM 偏移的平行轴项）
S[i]          : (n, 6)     float   运动子空间（rev/pris 只有一列）
levels[]      : list[list[int]]    预排的层级索引
```

### Per-env 动态状态

```
q      : (N, nq)
qdot   : (N, nv)
tau    : (N, nv)
```

### 中间变量

```
X_J_R   : (N, n, 3, 3)     关节变换旋转
X_J_r   : (N, n, 3)        关节变换平移
X_up_R  : (N, n, 3, 3)     复合变换 X_tree @ X_J
X_up_r  : (N, n, 3)
X_w_R   : (N, n, 3, 3)     世界变换
X_w_r   : (N, n, 3)
v       : (N, n, 6)        body 空间速度
c       : (N, n, 6)        bias 加速度
IA      : (N, n, 6, 6)     关节空间惯量
pA      : (N, n, 6)        bias 力
U       : (N, n, 6)        IA @ S（单 DOF 关节退化为 6-vec）
d_inv   : (N, n)            S^T @ U 的逆（标量，因为 nv=1）
u       : (N, n)            tau_i - S^T @ pA（标量）
a       : (N, n, 6)        空间加速度
f_ext   : (N, n, 6)        外力（contact + self-collision 合并）
qddot   : (N, nv)          广义加速度输出
```

关键简化：四足所有 actuated joint 都是 nv=1 的 revolute，FreeJoint 只有 root 一个。因此 U 是 6-vector 而不是 6xnv 矩阵，D 是标量不需要矩阵求逆。

## 3. 逐步计算图

### Step 0: Passive torques

```
并行维度: N x n_actuated
读: q[e, q_idx], qdot[e, v_idx], joint params
算: limit_torque (2 比较 + 2 乘 + 1 max) + damping (1 乘)
写: tau_passive[e, v_idx]
无依赖，无同步。~10 FLOP x N x n_actuated
```

### Step 1: Joint transforms

```
并行维度: N x n
REVOLUTE: cos, sin + Rodrigues (c*I + s*skew(k) + (1-c)*k*k^T) ~45 FLOP
FREE: quat_to_rot ~27 FLOP + pos copy
FIXED: identity / stored offset
PRISMATIC: R=I, r = axis * q[0]
无依赖，无同步。
```

### Step 2: Composite transform X_up = X_tree @ X_J

```
并行维度: N x n
X_up_R = X_tree_R @ X_J_R        # 3x3 matmul: 45 FLOP
X_up_r = X_tree_r + X_tree_R @ X_J_r  # matvec + add: 15 FLOP
无依赖，无同步。可与 Step 1 合并。
```

### Step 3: Forward pass（FK + velocity + IA/pA 初始化）— 按层执行

```
for level = 0, 1, ..., L-1:     # L 次 kernel launch
  并行维度: N x |level|
  每个 thread (e, i):

    --- FK compose ---
    if parent < 0:
      X_w = X_up
    else:
      X_w_R = X_w_R[parent] @ X_up_R    # 3x3 matmul: 45 FLOP  (读 parent，上一层已写完)
      X_w_r = X_w_r[parent] + X_w_R[parent] @ X_up_r  # 15 FLOP

    --- Velocity propagation ---
    E = X_up_R^T, r = X_up_r
    apply_velocity: E @ (v_parent_lin + cross(omega_parent, r)), E @ omega_parent
    + S * qdot_i  ~60 FLOP

    --- Bias acceleration c[i] ---
    c = spatial_cross_velocity(v) @ vJ  ~27 FLOP (FixedJoint 跳过)

    --- IA/pA init ---
    IA[e,i] = I_mat[i]                    # 36 float copy
    Iv = I_mat[i] @ v[e,i]               # 6x6 matvec: 66 FLOP
    pA[e,i] = spatial_cross_force(v, Iv)  # ~27 FLOP

  ====== SYNC（层间 barrier）======
```

### Step 4: Contact forces

```
并行维度: N x n_contacts
pos_world = R @ contact_local + origin    # 15 FLOP
vel_world = R @ v_lin + cross(R @ omega, r_arm)  # ~30 FLOP
F_normal = max(0, k*depth - b*vz)
F_tangent = -mu * F_n * v_slip / sqrt(|v_slip|^2 + eps^2)
Transform to body frame via X_world.inverse()
ATOMIC ADD: f_ext[e, body_idx] += spatial_force  # 6 floats
```

### Step 5: Self-collision forces

```
并行维度: N x n_pairs
AABB overlap test (6 min/max + 3 compare)
MTV penalty force + optional damping
ATOMIC ADD: f_ext[e, bi] and f_ext[e, bj]  # 2 x 6 floats
Step 4 和 Step 5 可并行执行。
====== SYNC ======
```

### Step 5.5: Merge ext_forces into pA

```
pA[e,i] -= f_ext[e,i]    # element-wise, N x n x 6
```

### Step 6: ABA Pass 2（backward, leaf -> root）— 按层执行

```
for level = L-1, ..., 0:     # L 次 kernel launch
  并行维度: N x |level|
  每个 thread (e, i):

    --- 对 nv=1 关节（绝大多数）---
    U = IA @ S             # 6x6 matvec: 66 FLOP
    d = S^T @ U            # dot: 11 FLOP
    d_inv = 1/d            # 1 div
    u = tau - S^T @ pA     # dot: 11 FLOP + 1 sub

    IA_A = IA - d_inv * outer(U, U)   # 对称 rank-1 update: ~57 FLOP
    pA_A = pA + IA_A @ c + U * (d_inv * u)  # ~78 FLOP

    --- 对 FreeJoint (nv=6, 只有 root) ---
    S = I_6, U = IA, D = IA
    D_inv = inv(IA)        # 6x6 矩阵求逆 (Cholesky)

    --- 向 parent 传递 ---
    # Congruence transform: IA[parent] += X^T @ IA_A @ X
    #
    # X 有块三角结构:
    #   X = [ E,  -E@skew(r) ]    E = R^T (3x3)
    #       [ 0,   E         ]
    #
    # 可分块展开为 ~8 个 3x3 matmul (vs 朴素 2 个 6x6 matmul)
    # 分块: ~360 FLOP (vs 朴素 ~792 FLOP)

    ATOMIC ADD: IA[e, parent] += X^T @ IA_A @ X    # 36 floats
    ATOMIC ADD: pA[e, parent] += apply_force(pA_A)  # 6 floats

    # Atomic 冲突: 同层 siblings 共享 parent。四足最多 4 路冲突，可接受。

  ====== SYNC（层间 barrier）======
```

**Pass 2 是计算最密集的环节。**

### Step 7: ABA Pass 3（forward, root -> leaf）— 按层执行

```
for level = 0, ..., L-1:     # L 次 kernel launch
  并行维度: N x |level|
  每个 thread (e, i):
    if parent < 0:
      a_p = X_up.apply_velocity(-gravity)   # ~5 FLOP
    else:
      a_p = X_up.apply_velocity(a[parent])  # ~33 FLOP

    if nv == 1:
      qddot = d_inv * (u - dot(U, a_p + c))  # ~14 FLOP
      a = a_p + c + S * qddot                 # ~12 FLOP
    elif nv == 6 (FreeJoint):
      qddot = D_inv @ (u - IA @ (a_p + c))   # ~132 FLOP

  ====== SYNC（层间 barrier）======
```

### Step 8: Integration

```
并行维度: N x n_bodies
Semi-implicit Euler:
  qdot_new = qdot + dt * qddot
  q_new = q + dt * qdot_new (用更新后的 qdot)
  FreeJoint: 四元数积分 + normalize
无依赖，无同步。
```

```

                              ==== SYNC ======+

```

## 5. 运算热点分析 (N=4096, n=17)

| 环节 | 核心运算 | per-thread FLOP | 并行度 | 总 FLOP |
|------|---------|----------------|--------|---------|
| Pass 1 forward | 3x3 matmul, matvec, cross, 6x6 matvec | ~200 | N x 17 | 14M |
| **Pass 2 backward** | **congruence (3x3 blocks), outer, dot** | **~500** | N x 17 | **35M** |
| Pass 3 forward | dot, matvec | ~50 | N x 17 | 3.5M |
| Contact | matvec, cross | ~100 | N x 4 | 1.6M |
| Integration | scalar + quat | ~30 | N x 17 | 2M |

## 6. Memory-bound 分析

```
每 body 每 env 读写:
  IA: 36 float (R+W) = 288 B
  pA: 6 float = 48 B
  v, c, U, u, d_inv: ~96 B
  X_up (R+r): 48 B
  总计: ~480 B per body per env

17 bodies x 4096 envs x 480 B = 33 MB per step
A100 HBM bandwidth: 2 TB/s -> 16.5 us 理论极限
Compute: 50M FLOP / 19.5 TFLOPS = 2.6 us 理论极限

结论: memory-bound (~6x)
```

## 7. Tensor Core + TMA 分析

### 核心观点

问题是 memory-bound，但在 Hopper 架构上，不走 TC+TMA 流水线带宽也跑不满。TMA 是异步 bulk copy (HBM -> SMEM)，TC (wgmma) 从 SMEM 消费。两者配合才能达到峰值带宽。

### 数据布局（为 TMA 优化）

env 维度必须是连续的内存维度，以便 TMA 做 2D tile 加载：

```
IA:   (n_bodies, 6, 6, N)   # env 轴最内层，TMA tile 沿 env 切
pA:   (n_bodies, 6, N)
v:    (n_bodies, 6, N)
q:    (nq, N)
qdot: (nv, N)
```

### 操作分类：哪些能走 TC

**能走 TC 的（至少一个操作数 env-shared）：**

| 操作 | GEMM 形状 | 说明 |
|------|-----------|------|
| X_tree @ X_J | (3,3) @ (3,N) | X_tree 常量 |
| I_mat @ v (pA init) | (6,6) @ (6,N) | I_mat 常量 |
| S^T @ pA | (1,6) @ (6,N) | S 常量 |
| FreeJoint inv(IA) | batched Cholesky | trsm 可走 TC |

约占 30-40% 总 FLOP。

**不能直接走 TC 的（两侧都 per-env）：**

| 操作 | 说明 |
|------|------|
| FK compose: X_w[parent] @ X_up | 3x3, 两侧 per-env |
| Congruence: X^T @ IA_A @ X | 6x6 分块为 3x3, 两侧 per-env |
| apply_velocity / apply_force | 3x3 matvec, 两侧 per-env |

原因分析：
- Batched 3x3 matmul, pad 到 8x8 后利用率 27/512 = 5%, TC 反而亏
- Pad 到 4x4 后 wgmma 最小 k=8 (TF32) 仍不满足
- N 个 env 的同一 3x3 matmul 无法折叠为单个大 GEMM（两个操作数都 per-env varying，Hadamard product 不是 GEMM）

约占 50-60% 总 FLOP。

### 对 per-env varying 操作的策略

当前结论：CUDA core + TMA（不走 TC），利用 TMA 异步搬运 + 多 stage 软件流水遮盖延迟：

```
stage k:   TMA async load tile_{k+2} -> SMEM buffer 2
stage k:   CUDA core compute on tile_k from SMEM buffer 0
```

## 8. 悬而未决的问题

以下问题尚未得到解答，需要后续深入讨论：

### 算法层面

1. **per-env varying 的 batched 小矩阵乘是否有更好的 TC 映射方案？**
   - 当前结论是 3x3 / 6x6 太小无法高效走 TC，但用户认为"还有很多没考虑到的"
   - 可能方向: 改变计算结构（非 Featherstone 的替代算法？）、不同的矩阵分块方式、混合精度策略

2. **congruence transform X^T @ IA @ X 是否有更好的分解？**
   - 当前方案: 分块为 ~8 个 3x3 matmul + cross product
   - 是否存在利用 IA 对称性或 X 块三角性质的更优分解？
   - 是否可以把 congruence 重写为一系列 rank-update 操作？

3. **是否应该考虑非 Featherstone 的算法？**
   - 比如 composite rigid body algorithm (CRBA) 计算 M 矩阵后 solve M @ qddot = tau - C
   - M 是 (nv, nv) 矩阵, 对四足 nv~18+6=24, 这个矩阵足够大可以走 TC
   - 但 CRBA 是 O(n^2), 且 solve 也有开销

4. **IA 的对称性利用**
   - 6x6 对称阵只有 21 个独立元素
   - 对称存储是否影响 TC/TMA tile 对齐？
   - 如何在 congruence transform 中保持对称性？

5. **float32 精度是否足够？**
   - Pass 2 的 d = S^T @ IA @ S 如果数值范围大（轻 foot vs 重 torso）可能丢精度
   - 需要实测

### 工程层面

6. **Atomic add 方案选择**
   - Pass 2: 同层 siblings 共享 parent, 最多 4 路冲突
   - 方案 A: 直接 36 个 atomic_add（低冲突可接受）
   - 方案 B: 预分配 per-child buffer + parent reduce（无 atomic）
   - 方案 C: warp shuffle reduction
   - 对人形等更复杂拓扑，哪个方案更通用？

7. **kernel fusion 策略**
   - 同一层的多个操作（FK + velocity + IA init + pA init）fusion 进一个 kernel？
   - 中间变量留在 register/SMEM，减少 global memory round-trip
   - fusion 的粒度如何选择？

8. **level barrier 的实现**
   - 多次 kernel launch（当前 ~20 次/step）的 launch overhead 是否成为瓶颈？
   - 替代方案: cooperative groups grid-level sync？persistent kernel？
   - CUDA graph capture 减少 launch overhead？

9. **CUTLASS 3.x grouped GEMM 对小矩阵的实际表现**
   - batch=4096, M=N=K=8 (padded from 6) 的 throughput 是多少？
   - 是否比手写 register-tiled CUDA core kernel 更快？
   - 需要 benchmark

10. **Warp 框架 vs 原生 CUDA**
    - Warp 是否支持 TMA 和 wgmma intrinsics？
    - 如果不支持，是否需要关键路径用原生 CUDA + 其余用 Warp？
    - 或者完全用原生 CUDA / CUTLASS？
